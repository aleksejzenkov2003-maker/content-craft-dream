
Цель: убрать текущее зависание на «Загрузка FFmpeg 5%» и сделать отмену/повтор предсказуемыми даже при плохой сети.

1) Подтверждённая причина зависания
- Да, я понимаю проблему: `fetchWithProgress()` блокирует поток (`response.body.getReader()`), а при отсутствии `content-length` пытается вызвать `response.blob()`, что даёт ошибку `body stream is locked`.
- Дополнительный фактор: в `getSharedFFmpeg()` ожидание уже запущенного `loadingPromise` не учитывает `AbortSignal`, поэтому кнопка отмены не прерывает ожидание, если загрузка была запущена фоном (`preloadFFmpeg`).

2) Исправления в `src/lib/ffmpegLoader.ts`
- Переписать загрузку core на единый путь без `body stream is locked`:
  - не читать `reader`, если потом нужен `blob()`;
  - либо использовать `response.clone()` для ветки без прогресса.
- Добавить `fetchWithTimeout()` + объединение сигналов (внешний `AbortSignal` + внутренний timeout), чтобы не висеть бесконечно на JS/WASM.
- Убрать «немой» `toBlobURL(...)` для `ffmpeg-core.js` и загружать JS так же, как WASM (с таймаутом/abort и контролируемым прогрессом).
- Сделать `await loadingPromise` прерываемым:
  - если пришёл abort от UI, бросать `AbortError` для этого вызова, не блокируя интерфейс;
  - фоновая предзагрузка может продолжаться отдельно.
- Очистка:
  - гарантированный `clearTimeout/clearInterval`;
  - корректный reset `loadingPromise` после завершения/ошибки;
