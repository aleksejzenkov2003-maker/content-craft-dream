
Исправлю склейку как задачу на “корректность важнее скорости”, потому что сейчас проблема не в одном баге, а в самом подходе.

## Что сейчас ломает результат

В `src/lib/videoConcat.ts` склейка идёт по “сырым” входным файлам:
- `main`, `back cover` и `intro` подаются в concat без полной предварительной нормализации
- `-r 30` и `-s 1080x1920` сейчас стоят на выходе, а не приводят каждый сегмент к одному профилю до склейки
- интро создаётся в `15 fps`, а остальное потом собирается в `30 fps`
- нет гарантии одинакового аудио-профиля у всех кусков
- если у задней обложки нет аудио или оно отличается по sample rate / channel layout, появляются ускорение, рассинхрон и “ломаются” уже вшитые субтитры визуально

Отдельно вижу, что есть старый `supabase/functions/concat-video/index.ts`, который тоже работает через низкоуровневую MP4-склейку без нормализации. Его использовать для решения этой проблемы не стоит.

## Что буду менять

### 1. Перед склейкой нормализовать каждый сегмент отдельно
Файл: `src/lib/videoConcat.ts`

Сделаю двухэтапный пайплайн:

```text
оригиналы
  -> normalize main
  -> normalize back cover
  -> normalize intro
  -> final concat
```

Для каждого сегмента приведу к одному формату:
- 1080x1920
- 30 fps
- yuv420p
- AAC
- 48 kHz
- stereo
- старт таймкодов с нуля

Это уберёт смену скорости и аудио-дрифт.

### 2. Интро из обложки делать сразу в том же профиле
Файл: `src/lib/videoConcat.ts`

Сейчас интро создаётся отдельно и в другом fps. Переделаю так, чтобы интро сразу рендерилось в тот же профиль, что и основное видео:
- 30 fps
- 48 kHz AAC
- такой же размер и pixel format

### 3. Обработать кейс “у задней обложки нет аудио”
Файл: `src/lib/videoConcat.ts`

Если у back cover нет звука, буду добавлять silent audio track через `anullsrc`, чтобы финальный concat всегда работал со стабильной парой `video + audio`, а не ломал таймстампы.

### 4. Убрать текущую “склейку по сырому входу”
Файл: `src/lib/videoConcat.ts`

Заменю текущую прямую сборку:
- либо полностью уберу текущий concat по необработанным входам
- либо оставлю только как внутреннюю стадию после нормализации, но не на исходных файлах

Идея: больше не пытаться склеивать разнородные MP4 напрямую.

### 5. Добавить валидацию результата перед загрузкой
Файл: `src/lib/videoConcat.ts`

После финального рендера добавлю проверки:
- файл реально создан
- размер не пустой
- есть выходной поток
- при ошибке не загружать “битый, но формально существующий” mp4

Это важно, потому что сейчас проверка по размеру слишком слабая.

### 6. Сохранить текущую UX-логику статусов, но сделать ошибки понятнее
Файл: `src/hooks/useVideoConcat.ts`

Оставлю текущие статусы `concatenating / checked / needs_concat`, но уточню сообщения об ошибках, чтобы было видно:
- не удалось нормализовать основной ролик
- не удалось нормализовать back cover
- отсутствует/битое аудио
- не удалось собрать финальный файл

## Какие файлы затрону

- `src/lib/videoConcat.ts` — основное исправление пайплайна
- `src/hooks/useVideoConcat.ts` — более точная обработка ошибок и статусов
- при необходимости `src/components/publishing/PublicationsTable.tsx` — только если потребуется показать более понятный текст статуса, без изменения логики запуска

## Технически

Нормализация будет в духе такого профиля:

```text
video:
scale=1080:1920:force_original_aspect_ratio=decrease,
pad=1080:1920:(ow-iw)/2:(oh-ih)/2,
fps=30,
format=yuv420p,
setpts=PTS-STARTPTS

audio:
aresample=48000:async=1:first_pts=0,
aformat=sample_rates=48000:channel_layouts=stereo,
asetpts=PTS-STARTPTS
```

Для сегментов без звука:
```text
anullsrc=r=48000:cl=stereo
```

То есть исправление будет не “подкрутить один флаг”, а перевести склейку на жёстко нормализованный монтажный пайплайн. Это должно устранить:
- изменение скорости ролика
- отставание/опережение озвучки
- визуально сломанные субтитры после склейки
